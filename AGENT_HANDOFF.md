### Обогащение стояло 2 дня — починено
- **Симптом:** enrich каждые 5 мин падал `enriched:0, blocks:15, ABORT`. Застряло на 55% (88k/160k).
- **Причина (докопались тестами):** rekvizitai за **Cloudflare**. Cloudflare **забанил IPv4 VPS** (`178.104.253.76` → 403)
  после 2 дней долбёжки в 10 потоков. IPv6 хоста (`2a01:4f8:c2c:9500::1`) **чист (200)**. Хост по умолчанию
  ходит по IPv6 (потому curl с хоста давал 200 и путал нас), а **контейнер имел только IPv4** → каждый запрос 403.
  Это НЕ undici, НЕ потоки, НЕ код — это бан IPv4 на уровне Cloudflare (даже curl из контейнера по IPv4 = 403).
- **Фикс (коммиты до `8cac0f1`):**
  - `workers/enrich/index.ts`: движок фетча с undici `fetch` → **`node:https` с опцией `family`**; rekvizitai-фетчи
    идут **строго по IPv6 (`family=6`)**, прочие (домен-угадка/email) — dual (иначе IPv4-only сайты отвалятся).
  - `docker-compose.yml`: сеть `internal` += `enable_ipv6: true` + ULA-подсеть `fd00:1ead:9e0::/64` (NAT66).
    Docker 29 имеет `ip6tables` по умолчанию → NAT66 на глобальный IPv6 хоста заработал без правки daemon.json.
  - `CONCURRENCY=10` (откатывал на 6, но пользователь решил оставить 10 — добить остаток быстрее), cron `--limit=800`.
- **Результат:** ручной прогон `enriched:20/20, blocks:0`. База добивается сама ~85 комп/мин, ~14ч до 100%.
- ⚠️ **IPv6 — последний чистый IP.** Если Cloudflare забанит и его — чистых выходов не останется (только прокси/переезд).
  Пользователь это принял (осталось <45%, должны проскочить).

## ⚡ ПЕРЕСМОТР СТРАТЕГИИ ОБОГАЩЕНИЯ (главное за вечер 2026-06-04)

На реальных данных выяснилось, что **Google Places слаб для литовского B2B** (76% компаний
`not_in_places`, частые неверные матчи, имя без города не различает тёзок). Развернулись:

### Новый основной источник обогащения = rekvizitai.vz.lt (по КОДУ компании)
- **Поиск по коду** (надёжно, ~100% находит, без слаг-угадывания):
  `https://rekvizitai.vz.lt/imones/1/?...&company_code=<rc_code>&...&redirected=1` → в ответе
  `Rasta įmonių: N` + ссылка `/imone/<slug>/` → фетчим карточку.
- **С карточки за один проход:** сайт (`Tinklalapis`), телефон+мобильный, **кредит-риск**, адрес+город.
- **БАН: проверено — нет.** С датацентрового IP VPS прогнали 824 + 80-бёрст + 12k ночной — **0 блоков**.
  Старый бан был на СТРАНИЦАХ-СПИСКАХ (пагинация), а карточки `/imone/` открыты.

### Кредит-риск (Kreditavimo rizika) = ГЛАВНЫЙ фильтр мусора
Шкала A→E (Žemiausia/Žema/Vidutinė/Aukšta/Aukščiausia), на карточке (цвет рамки + подпись).
**Решение пользователя: оставляем A/B/C, режем D/E.** Это главный сигнал «живая/дохлая»,
сильнее выручки. Доступен ТОЛЬКО парсингом rekvizitai (в реестре его нет).

### Финансы (выручка/прибыль) = атрибут карточки, из API реестра БЕСПЛАТНО
`get.data.gov.lt/.../jar/pelno_ataskaitos/PelnoAtaskaita?juridinis_asmuo.ja_kodas=<code>` —
traversal по коду работает. Берём последний год: `PARDAVIMO PAJAMOS` (выручка) + строка `GRYNAS...PELN/NUOSTOL` (прибыль).

### ⚠️ ВАЖНО: rekvizitai НЕДОСЧИТЫВАЕТ сайты — нужна домен-верификация
Пустое поле `Tinklalapis` ≠ «нет сайта». Проверка: из 12 «без сайта по rekvizitai» компаний €500k–2M
**7 реально имеют сайт** (conpre.com, valsva.lt, westcapitalgroup.com, texcup.lt, daitas.com, se.lt, digizent.com) —
просто не внесли в rekvizitai. Реальный «нет сайта» для крупных ≈ ~20%, а не 50%.
**В воркер ОБЯЗАТЕЛЬНО добавить шаг:** домен по имени `{slug}.lt/.com/.eu` + верификация (rc_code или имя на странице,
редирект на rekvizitai = не сайт). Только после этого классифицировать ветку A (сайт есть) / B (реально нет).
Дёшево (пара HTTP-фетчей, без квот). Иначе «нет сайта»-лиды будут ложными (позор: «у вас нет сайта» — а он есть).
Перепрогнать «no-site» из ночного результата с этой проверкой → честный site/no-site.

### Email — разрозненный, НЕ из реестра
В официальном реестре (data.gov.lt) контактов НЕТ вообще (только код/имя/адрес/финансы/форма).
rekvizitai email не показывает. Реально берётся: **с сайта самой компании** (`/kontaktai`, ~85% у кого сайт есть).
У no-site компаний почты почти нет → они **телефонные лиды**. (scoris=Cloudflare-блок; visalietuva=cf-обфускация; ltkatalogas=по слагу — не масштабно.)

### Стратегический сдвиг продукта (КЛЮЧЕВОЕ)
Сервис ≠ «робот холодной рассылки». Это **движок КАЧЕСТВЕННЫХ лидов**:
- Отдаём 15–20 лидов/день → они должны быть НЕ мусорными. Покрытие email не важно, важно качество.
- 160k ÷ 20/день = годы запаса → можно быть беспощадно придирчивым.
- **Quality-фильтр = кредит-риск (A/B/C) + выручка ≥ порог** → ~37% премиум (оценка) ≈ ~60k лидов.
- Два канала по ветке: **A (плохой сайт)** → авто-письмо+аудит (email с их сайта); **B (нет сайта)** → звонок «сделаю сайт».
- Tier 1–2 email не нужен вообще (отдаём лид с телефоном). Email/авто-письмо — фича Tier 3 для сайт-сегмента.
