# Архитектура — Leadgen LT

Высокоуровневая логика. Бизнес-обоснования — `leadgen-service-konspekt.md`. Реализация — `leadgen-techspec.md`.

## Воронка (4 слоя)

```
Слой 1: RC API → Postgres        забор ТОЛЬКО действующих (veikiantis=1, ~160k)
   ↓
Слой 2: фильтр по нише           WHERE evrk2_code LIKE '41%'  (локально, бесплатно, индекс)
   ↓
Слой 3: обогащение сайтом        Google Places (1 запрос) + HTML-верификация
   ↓
Слой 4: аудит + решение          PageSpeed → ветка A / ветка B
```

### Ветка A — сайт есть
Дизайн → скорость → аудит проблем.
- Проблем нет → `recheck_later` (отложенные, перепроверить через N мес.)
- Проблемы есть → **лид**

### Ветка B — сайта нет, компания работает
(FB/Etsy/маркетплейс) → **лид** (предложить сайт с нуля)

## Backend-модули (воркеры, Node + node-cron)

| Модуль | Что делает | Cron |
|---|---|---|
| `rc-sync` | забор активных компаний + классификатор EVRK → `companies` | нед./мес. |
| `enrich` | 1 Places-запрос → кандидаты → scoring → HTML-верификация → PageSpeed | по нишам, под бюджет 5000/мес |
| `enrich-fallback` | нерешённые (`ambiguous`/нет кандидата) → Bing/Brave/rekvizitai | по очереди |
| `audit-gen` | шаблонный аудит (LLM только Tier 3) | после enrich |
| `email-gen` | GPT-4o литовское письмо (фаза 2, Tier 3) | по лиду |
| `lead-select` | дневной отбор по тарифу + дедупликация | ежедневно |
| `recheck` | перепроверка отложенных, миграция между вёдрами | по `recheck_at` |

## Обогащение (Слой 3) — ПЕРЕСМОТРЕНО 2026-06-04

> На живых данных Google Places оказался слаб для LT B2B (76% `not_in_places`). **Основной источник —
> rekvizitai.vz.lt по коду компании** (карточка → сайт + телефон + кредит-риск + адрес), финансы — из API реестра,
> email — с сайта компании. Кредит-риск (A/B/C keep, D/E drop) = главный фильтр мусора. Эндпоинты и детали —
> в `docs/API_RC.md` и `AGENT_HANDOFF.md`. Продукт = движок КАЧЕСТВЕННЫХ лидов (15–20/день), не «рассылка всем».
> Раздел про Google Places ниже оставлен как вторичный вариант (для ниш с витриной).

### Старый вариант — Google Places (вторичный)

**Google Places = источник КАНДИДАТОВ, не финальная истина.**
- ГЛАВНОЕ ПРАВИЛО: **1 компания = максимум 1 Text Search запрос**.
- НЕ делать 3–5 вариаций запроса — это сжигает месячный лимит.
- Точность повышаем: кандидаты из одного ответа + confidence-scoring + lightweight HTML-верификация.
- Жёсткий стоп на 5000 событий/мес — система сама останавливается, без доплат. Сброс 1-го числа.
- Не обработанные из-за лимита → `skipped_limit`, ждут следующего месяца.

**Lightweight HTML-верификация** (обычный HTTP fetch, не Puppeteer, не Places):
подтверждаем что сайт принадлежит компании по: RC-код на сайте / совпадение телефона / адреса / имя+город.

**Правила выдачи по уверенности:**
- `high` → `auto_approved`, выдаётся автоматически
- `medium` / `ambiguous` → `needs_review`, не выдаётся без проверки
- `low` / `wrong_match` → не выдаётся автоматически

## Динамика базы (самообновление) — три ведра
```
active_lead   — есть проблема → в работу
recheck_later — сайт норм, перепроверить через N мес. (триггер «проект стоит на месте»)
dead          — неактивные/закрытые
```
Сайты деградируют → часть «не наших» сами перетекают в «наши». Сток не иссякает:
follow-up + старение сайтов + приток новых компаний.

## Фазы

### Фаза 1 — личный инструмент
Backend-пайплайн (RC → активность → ниша → обогащение → лиды) + веб-дашборд (лиды, аудит, статистика).
**Архитектура уже мультитенантная** (`subscriber_id`), но активен один subscriber (я).
**PWA + push заложены сразу.** Первые ниши: стройка (EVRK 41–43) + недвижимость (68).

### Фаза 2 — подписка
Регистрация подписчиков, онбординг, тарифы, настройки (no-code), Resend подписчика,
дедупликация, биллинг, триал. Якорь — Tier 2 (аудит).

### Фаза 3+ — горизонт
SMM/SEO/дизайн направления (разный критерий «проблемы»), входящий канал/маркетплейс,
CRM в Tier 4 (мультитенантная SaaS).

## Воркфлоу подтверждения (Tier 3)
```
lead-select → audit-gen → email-gen (draft) → push «письмо готово»
   → подписчик открывает очередь → проверяет/правит → «Подтвердить отправку»
   → Resend подписчика (его домен, его имя) → статус sent
```
**Подтверждение человеком = юридический предохранитель (GDPR). Не делать полный автомат.**

## Граница ответственности (в оффер открыто)
- **Мои 20%**: целевой лид — активная компания, нужная ниша, реально нет/плохой сайт, свежий, с аудитом.
- **Их 80%**: обработка, продажа, цены, услуги, портфолио. «Лид» = отфильтрован по критерию, НЕ гарантия покупки.
