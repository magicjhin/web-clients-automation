# Architecture — Архитектура системы

## Воронка фильтрации

```
rekvizitai.lt
    │
    ▼
Фильтр 1: статус активная
Фильтр 2: нет налоговых долгов
Фильтр 3: сдаёт финансовую отчётность
Фильтр 4: работает 2+ лет
Фильтр 5: есть email
    │
    ▼
Есть сайт?
    │
    ├── НЕТ → статус: no_website → письмо тип 2 (создание сайта)
    │
    └── ДА → сайт доступен?
                │
                ├── НЕТ → статус: site_down → is_vip: true → письмо тип 3 (срочное)
                │
                └── ДА → PageSpeed score?
                            │
                            ├── > 60 → статус: rejected
                            │
                            └── ≤ 60 → скриншот → Claude оценивает дизайн
                                            │
                                            ├── дизайн норм → rejected
                                            │
                                            └── дизайн плохой → Google Maps активность?
                                                                    │
                                                                    ├── неактивный → rejected
                                                                    │
                                                                    └── активный → статус: qualified → письмо тип 1
```

## VIP система (scripts/filter/vip.js)

| Сигнал | Баллы |
|--------|-------|
| Сайт не работает | 100 |
| PageSpeed < 30 | +30 |
| Сайт до 2015 года | +20 |
| Нет мобильной версии | +15 |
| Нет SSL | +10 |
| Ниша с высоким чеком | +15 |
| Активный бизнес | +10 |

VIP score > 80 → Telegram 🔥 уведомление → рекомендация позвонить

## Три типа писем (на литовском языке)

**Тип 1 — Плохой сайт (редизайн/ребилд)**
- Конкретные проблемы из аудита
- Предложение решения
- CTA: созвон 15 минут

**Тип 2 — Нет сайта (создание с нуля)**
- Аргумент: теряете клиентов без сайта
- CTA: созвон 15 минут

**Тип 3 — Сайт не работает (VIP, срочное)**
- Короткое, срочное сообщение
- CTA: позвоните прямо сейчас

## Повторные письма

```
День 0 → first_email_sent_at записан, статус: sent
День 3-4 → нет ответа → отправить letter_second (короткое follow-up)
После второго письма → статус: ignored
```

## Обработка ошибок

**Продолжаем (клиентские ошибки):**
| Ошибка | Действие |
|--------|---------|
| Сайт недоступен | VIP статус, продолжаем |
| Email не существует | email_failed + call_required=true |
| PageSpeed не ответил | retry через 30 мин |
| Скриншот не сделался | пропустить, продолжить |

**Останавливаемся (системные ошибки) + Telegram алерт:**
- PostgreSQL упал
- Claude API упал
- VPS перегружен (CPU > 90%)

**Безопасный перезапуск:**
Колонка `processing_step` в таблице companies сохраняет точное место остановки. При перезапуске скрипт читает processing_step и продолжает с того шага.

Возможные значения processing_step:
- `pagespeed` — проверка PageSpeed
- `screenshot` — скриншот
- `design` — оценка дизайна Claude
- `activity` — проверка активности Google Maps
- `vip` — расчёт VIP score
- `crawl` — краулинг сайта
- `audit` — Claude аудит (первый вызов)
- `audit_review` — Claude перепроверка аудита (второй вызов)
- `letter` — генерация письма
- `email` — отправка письма
- `done` — завершено

## Двойной аудит (scripts/audit/claude.js)

Аудит выполняется двумя последовательными вызовами Claude API:

```
Шаг 1 — Первичный аудит (processing_step: audit)
    Входные данные: crawl текст + скриншот + данные компании
    Выходные данные: JSON { problems[], recommendations[], confidence, letter_type }
    Сохраняется в: companies.audit (промежуточный результат)

    ↓

Шаг 2 — Перепроверка (processing_step: audit_review)
    Входные данные: результат шага 1 + исходные данные
    Промпт: "Вот аудит сайта. Найди противоречия, пропущенные проблемы или
             завышенные/заниженные оценки. Исправь и верни финальный JSON."
    Выходные данные: скорректированный JSON + финальный audit_confidence
    Сохраняется в: companies.audit (финальный), companies.audit_confidence

    ↓

Если audit_confidence < 6 после перепроверки:
    → processing_error: "low_confidence"
    → статус: pending (повтор через 24 часа)
    → Telegram уведомление

Если audit_confidence ≥ 6:
    → processing_step: letter
    → генерация письма
```

**Зачем:** первый вызов иногда пропускает важные проблемы или преувеличивает мелкие. Второй вызов смотрит на результат свежим взглядом и выступает критиком — итоговое письмо получается точнее и убедительнее.

## n8n воркфлоу (общая схема)

```
Воркфлоу 1: Парсинг
Telegram /parse → Webhook → Execute Command → node scripts/parser/index.js --niche=X

Воркфлоу 2: Фильтрация
Trigger → Execute Command → node scripts/filter/index.js [--limit=50]

Воркфлоу 3: Аудит и письма
Cron 09:00 → Execute Command → node scripts/audit/index.js --batch=15

Воркфлоу 4: Повторные письма
Cron ежедневно → Execute Command → node scripts/email/followup.js

Воркфлоу 5: Telegram бот
Webhook (Telegram) → Switch по команде → Execute Command → ответ в Telegram

Воркфлоу 6: IMAP проверка
Cron каждые 30 мин → Execute Command → node scripts/email/imap.js
```
